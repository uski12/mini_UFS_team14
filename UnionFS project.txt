Cloud Computing: "Mini-UnionFS" 
Project 
1. Background & Motivation 
When you run a Docker container, it doesn't copy the entire base image. Instead, it stacks a 
read-write "container layer" on top of read-only "image layers." This allows multiple containers to 
share the same underlying base OS files without interfering with one another. 
To deeply understand this paradigm, you will not just be using containers—you will be building 
the file system mechanism that makes them possible. 
2. Project Objective 
Your task is to implement Mini-UnionFS, a simplified Union File System running in userspace 
using FUSE (Filesystem in Userspace). 
Your program will take multiple underlying directories (branches) and mount them as a single, 
unified virtual file system. It must successfully manage read-only lower layers and a read-write 
upper layer, accurately implementing the Copy-on-Write (CoW) strategy and handling file 
deletions via "whiteout" files. 
3. Core Requirements 
Your implementation must support the following features: 
● Layer Stacking (Directory Union): 
○ The file system must accept at least two directories as input: a lower_dir (read-only 
base image) and an upper_dir (read-write container layer). 
○ When listed, the mounted Mini-UnionFS should show a merged view. If a file exists 
in both, the upper_dir file takes precedence. 
● Copy-on-Write (CoW): 
○ If a user modifies a file that originally exists only in the lower_dir, copy the file to the 
upper_dir, apply modifications there, and leave the lower_dir file untouched. 
● Whiteout (Deletions): 
○ If a user deletes a file originating in the lower_dir, create a "whiteout" file in the 
upper_dir (e.g., deleting config.txt creates a hidden .wh.config.txt). 
○ The file system must interpret a whiteout file as an instruction to hide the underlying 
file from the user. 
● Basic POSIX Operations: 
○ Support getattr, readdir, read, write, create, unlink, mkdir, and rmdir. 
4. Technical Specifications & Deliverables 
● Languages Allowed: C, C++, Go, or Rust. 
● Environment: Standard Linux environment (e.g., Ubuntu 22.04 LTS).

1. Source code pushed to the university Git repository.

Appendix B: Automated Test Suite (test_unionfs.sh) 
Place this script in the same directory as your compiled FUSE binary and run it to verify your 
core logic. 
#!/bin/bash 
FUSE_BINARY="./mini_unionfs" 
TEST_DIR="./unionfs_test_env" 
LOWER_DIR="$TEST_DIR/lower" 
UPPER_DIR="$TEST_DIR/upper" 
MOUNT_DIR="$TEST_DIR/mnt" 
 
# Colors 
GREEN='\033[0;32m' 
RED='\033[0;31m' 
NC='\033[0m' 
 
echo "Starting Mini-UnionFS Test Suite..." 
 
# Setup 
rm -rf "$TEST_DIR" 
mkdir -p "$LOWER_DIR" "$UPPER_DIR" "$MOUNT_DIR" 
echo "base_only_content" > "$LOWER_DIR/base.txt" 
echo "to_be_deleted" > "$LOWER_DIR/delete_me.txt" 

 
$FUSE_BINARY "$LOWER_DIR" "$UPPER_DIR" "$MOUNT_DIR" 
sleep 1 
 
# Test 1: Visibility 
echo -n "Test 1: Layer Visibility... " 
if grep -q "base_only_content" "$MOUNT_DIR/base.txt"; then echo -e 
"${GREEN}PASSED${NC}"; else echo -e "${RED}FAILED${NC}"; fi 
 
# Test 2: Copy-on-Write 
echo -n "Test 2: Copy-on-Write... " 
echo "modified_content" >> "$MOUNT_DIR/base.txt" 2>/dev/null 
if [ $(grep -c "modified_content" "$MOUNT_DIR/base.txt" 2>/dev/null) 
-eq 1 ] && [ $(grep -c "modified_content" "$UPPER_DIR/base.txt" 
2>/dev/null) -eq 1 ] && [ $(grep -c "modified_content" 
"$LOWER_DIR/base.txt" 2>/dev/null) -eq 0 ]; then 
    echo -e "${GREEN}PASSED${NC}" 
else 
    echo -e "${RED}FAILED${NC}" 
fi 
 
# Test 3: Whiteout 
echo -n "Test 3: Whiteout mechanism... " 
rm "$MOUNT_DIR/delete_me.txt" 2>/dev/null 
if [ ! -f "$MOUNT_DIR/delete_me.txt" ] && [ -f 
"$LOWER_DIR/delete_me.txt" ] && [ -f "$UPPER_DIR/.wh.delete_me.txt" ]; 
then 
    echo -e "${GREEN}PASSED${NC}" 
else 
    echo -e "${RED}FAILED${NC}" 
fi 
 
# Teardown 
fusermount -u "$MOUNT_DIR" 2>/dev/null || umount "$MOUNT_DIR" 
2>/dev/null 
rm -rf "$TEST_DIR" 
echo "Test Suite Completed."
